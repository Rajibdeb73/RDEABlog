Prologue

The Architect's Dilemma

It is three in the morning in a bank's war room in Sydney. A single monitor glows in the dark. The timestamp reads 03:14. A payment has just failed — not crashed, not timed out. It made the wrong decision. The AI model, trusted with real-time fraud scoring on Australia's New Payments Platform rail, declined a legitimate payment from a loyal, verified customer. The customer called. Then complained. Then posted. The incident ticket is already open.

The architect on call stares at the logs. There is an output — DECLINED — but the reasoning is gone, stripped in a pipeline optimisation three weeks ago. The prompt version that ran? Unknown. Which retrieval chunks informed the decision? Not stored. Was the guardrail even active at that hour? Nobody is certain.

This is not a story about AI failing. It is a story about the absence of patterns.

Design patterns in software engineering are not academic constructs. They are the scar tissue of production systems — lessons encoded by practitioners who learned them under load, under pressure, with real consequences for getting it wrong. In AI systems deployed in regulated institutions, the stakes are higher still. A hallucinated policy answer is a potential consumer harm event. A prompt injection in an internal tool is a data breach. An unversioned system prompt is an unexplainable decision. An agent without a stopping condition is a production incident.

This guide is organised as a journey through four layers of patterns. Each builds on the one beneath it. Together, they form the architecture of an AI system that a bank's risk committee, its regulators, and its architects can all trust — not merely in a demonstration environment, but at midnight on a Monday, under load, with real money and real customers.

The patterns that follow are the answer to that war room question. They are what was missing.

Layer One · Foundation Patterns

Before the agent can coordinate,
it must first be able to think.

Six patterns that govern how a single agent reasons, retrieves, acts, and maintains state. These are the atoms. Get them wrong and no amount of orchestration sophistication will save the system.

Enterprise Principle
Get Layer 1 right before scaling to Layer 2. Most production failures trace back to a poorly designed foundation — an agent that hallucinates because it has no grounding, or loops because it has no stopping condition. Fix the primitive before adding orchestration complexity.
01

ReAct — Reason + Act

The closed-loop reasoning primitive
Core Pattern

ReAct is the workhorse of agentic AI. It gives a model the ability to alternate between thinking and acting — not as separate phases, but as an interlocked loop where each observation changes what the agent thinks next.

The name combines Reasoning with Action. The model produces a structured thought — "what do I know, what do I still need?" — then calls a tool or API. The response becomes an observation, feeding directly into the next reasoning step. This loop continues until a terminal condition is met.

What makes ReAct powerful in enterprise AI is not just its flexibility, but its traceability. Unlike a single-shot prompt, ReAct externalises the entire decision chain. Every thought is recorded. Every tool call logged. Every observation shapes a visible reasoning step. In regulated environments, this is not merely a quality improvement — it is an audit trail.

Banking scenario: KYC enrichment agent
ReAct Loop — KYC EnrichmentThought: Customer C-4821 is new. I need to verify sanctions status before proceeding. Action: sanctions_check({ entity_id: "C-4821", lists: ["OFAC","UN","AUSTRAC"] }) Obs: CLEAR — no matches across all three screening lists. Thought: Sanctions clear. Now I need PEP status and adverse media. Action: pep_lookup({ name: "John Smith", dob: "1975-03-14", country: "AU" }) Obs: MATCH — former regional councillor, Queensland 2014-2018. Tier 2 PEP. Thought: Tier 2 PEP match requires EDD and senior relationship manager sign-off. Action: adverse_media_scan({ entity: "John Smith", date_range: "2014-present" }) Obs: No adverse media findings in credible sources. Thought: PEP match without adverse media. Per policy 4.2: EDD queue with senior sign-off. Final: Risk tier = HIGH. Action = EDD_REQUIRED. Assignee = relationship_manager_pool.

This chain replaces a brittle rules engine with adaptive, observable reasoning. If the sanctions API returns an error, the agent records that observation and can escalate — rather than silently defaulting to CLEAR. Every decision step is reviewable by compliance, auditable by APRA, and explainable to the customer.

⚠ Critical Mistake — No Loop Termination Condition
Teams deploy ReAct agents without defining when the agent should stop. The agent continues calling tools — checking additional databases, requesting more context — long after sufficient information has been gathered. A 40-step KYC loop on a straightforward corporate customer is not intelligence; it is an operational incident surfacing in your APRA CPS 230 risk log. Beyond cost, runaway loops can trigger rate limits on downstream APIs and create cascading failures.
✓ The Fix
Encode explicit stopping criteria in the system prompt: a maximum number of steps (6–10 for KYC-style tasks) and a set of terminal actions that halt the loop (e.g., submit_risk_verdict, escalate_to_human). Enforce these in your runtime wrapper — not just in the prompt. If the agent reaches max_steps without a terminal action, the runtime automatically escalates to a human queue.

02

Chain of Thought

Stepped, visible reasoning — and your compliance audit trail
Audit-Critical

Chain of Thought forces the model to externalise every reasoning step before producing a final answer. In consumer contexts this improves accuracy. In regulated financial services, it does something more important: it creates the explainability artefact that regulators require.

The mechanics are straightforward. You instruct the model — either through a simple directive ("think step by step") or through worked examples — to show its reasoning before concluding. Three practical variants: Zero-shot CoT appends "Think step by step" to the user prompt. Few-shot CoT provides 2–3 worked examples with full reasoning chains. Process-of-Thought (PoT) instructs the model to emit pseudocode for numerical reasoning — particularly valuable in financial calculations where arithmetic accuracy is non-negotiable.

Banking scenario: credit decision with explainability
CoT credit decision — system instruction + model output// System instruction: "Reason through: (1) income stability, (2) DTI ratio vs 35% threshold, (3) repayment history over 48 months, (4) collateral and LVR vs 80% maximum. Show your reasoning for each step before concluding." Step 1 — Income: Salary $110k, same employer 3y 4m. Consistent pay stubs. STABLE. Step 2 — DTI: Existing debt $1,800/mo. Gross $9,167/mo. DTI = 19.6%. PASS. Step 3 — History: 2 late payments in 48 months, both 2022. No defaults. MINOR FLAG. Step 4 — Collateral: Property $620k. Loan $420k. LVR = 67.7%. Policy max 80%. ADEQUATE. Decision: APPROVE at standard variable rate. Conditions: Flag for 12-month repayment monitoring. Confidence: 0.89

This output is not merely a better decision — it is a defensible decision. When a customer lodges a dispute under the bank's IDR process, the reasoning chain is the evidence. When APRA examines automated credit decisioning, the chain of thought is the explainability artefact. When a model version changes and decisions shift, the chain reveals exactly which step changed and why.

⚠ Critical Mistake — Stripping CoT from the Audit Log
A developer, optimising API response size, strips the reasoning steps before logging — keeping only the final decision. When the compliance team audits automated credit decisions, all they have are APPROVE/DECLINE outcomes with no reasoning. Under APRA CPS 230, the inability to reconstruct and explain an automated decision is a model governance gap. Under ASIC's RG 271, it may be a consumer protection obligation failure.
✓ The Fix
Demand structured output that explicitly preserves reasoning: {"reasoning_steps": [...], "decision": "APPROVE", "confidence": 0.89, "flags": [...]}. Log the full structured object — not just the decision field. The reasoning steps are the compliance evidence. Treat them with the same retention discipline as the decision itself: seven years in Australia for regulated credit decisions.

03

Retrieval Augmented Generation

Grounding intelligence in verified, current knowledge
Critical Infrastructure

A large language model's knowledge is frozen at the moment of training. Your regulatory policies are not. RAG decouples knowledge from model weights — so your AI answers from your documents, not from memory that might be wrong, stale, or hallucinated.

RAG works in two phases. In the ingestion phase, your documents — policy manuals, APRA instruments, product disclosure statements — are chunked, converted into vector embeddings, and stored in a vector database. In the inference phase, the query is embedded using the same model, and the database is searched for the most semantically similar chunks. Those chunks are injected into context alongside the query. The model generates its answer grounded in those retrieved documents — not in training data.

The six-stage production RAG pipeline
RAG pipeline architectureStage 1 — Ingest: PDF/DOCX/HTML → parser → semantic chunker → embedder → vector DB Metadata: source, version, effective_date, document_class, jurisdiction Stage 2 — Embed query: user_query → same embedding model → query_vector (1536-dim) Stage 3 — Retrieve: vector_DB.similarity_search(query_vector, top_k=8, threshold=0.70) Stage 4 — Rerank: cross_encoder.rerank(query, candidate_chunks) → top_5 by relevance // eliminates false positives — critical for regulatory content Stage 5 — Augment: prompt = system_prompt + retrieved_chunks + user_query Stage 6 — Validate: faithfulness_check: every claim must be traceable to retrieved chunks

Note Stage 4 — the reranker. Vector similarity finds chunks that are semantically close, but semantic closeness is not the same as relevance. A reranker rescores candidates against the actual query, dramatically improving precision. For regulatory content where the difference between a clause that sounds relevant and one that is relevant can be legally significant, the reranker is not optional.

⚠ Critical Mistake #1 — Hard-Split Chunking Destroys Regulatory Clauses
A fixed 500-token hard split cuts a regulatory clause in half. The retrieved chunk reads "the bank may not engage in margin lending activities..." but the qualifying clause "...without prior approval from APRA under Section 23(b)" is in the next chunk, which was not retrieved. The model generates guidance that is the direct opposite of the actual regulatory requirement. In a compliance Q&A context, this is not an accuracy failure — it is a regulatory risk event.
✓ The Fix
Use semantic chunking that splits on natural document boundaries — section headings, paragraph breaks, numbered clause boundaries. Implement a sliding window overlap of 100–150 tokens at chunk boundaries. Store the document's hierarchical structure as metadata so when a clause chunk is retrieved, its parent section can be retrieved alongside it for full context.
⚠ Critical Mistake #2 — No Version Awareness
A 2022 version of an APRA prudential standard and the 2024 updated version both exist in the vector database. Without metadata-aware reranking, the model may answer based on the superseded version — citing a threshold that was increased in the 2024 revision. The answer is confident, fluent, and wrong.
✓ The Fix
Tag every document with effective_from, effective_to, and superseded_by metadata at ingestion time. Apply a recency boost in the reranking stage for regulatory content. For time-sensitive queries, filter the retrieval pool to documents whose effective_to date is null (currently in force).

Self-Reflection

The agent that challenges its own output before releasing it
Quality Gate

Self-Reflection gives an agent the ability to evaluate its own output against defined quality criteria before returning it to the caller. It is a compliance review built into the generation loop itself — without adding a human to every workflow.

The pattern has four stages: Generate a draft response. Critique it against a rubric — is every claim grounded in retrieved context? Does the output comply with the required format schema? Are regulatory references cited correctly? Revise if any criterion fails. And Gate at a maximum of three reflection rounds — after which unresolved issues escalate to human review.

In banking, Self-Reflection is the quality mechanism that makes AI suitable for compliance-adjacent tasks. A regulatory report self-reviewed against a structured rubric before human sign-off carries less risk than one that has not. A customer communication checked for accurate rate citations and appropriate hardship language before sending is less likely to trigger a complaint.

⚠ Critical Mistake — Same Context Window Critique
Placing the critique in the same context window as the generation creates confirmation bias. The model has already "committed" to its answer. When asked to critique it in the same context, it rationalises rather than genuinely challenges. For a compliance review task — where the entire value is in independent challenge — this produces false assurance rather than genuine quality improvement.
✓ The Fix
Use a separate API call for the critique step with a different system prompt: "You are a strict compliance reviewer. Your only job is to find errors, omissions, and inaccuracies. Be adversarial." In high-stakes flows — credit decisioning, AML reporting — use a structurally different model for the critique step. The critique call is a separate inference with no memory of the generation that produced the draft.

05

Memory Management

State across turns, sessions, and time
Stateful Systems

A stateless AI agent introduces itself to the same customer every single time. Memory Management is the architectural discipline that gives an agent continuity — across a single conversation, across sessions, and across the months-long lifecycle of a complex case.

Enterprise AI memory operates across three tiers. Tier One — In-context memory: the current conversation window. Fast, immediately available, completely ephemeral — gone when the session ends. Tier Two — External store: Redis, a vector database, or a relational store keyed by session ID or customer ID. Survives session boundaries. Enables an agent to pick up where a previous conversation ended. Tier Three — Episodic memory: compressed, summarised representations of past sessions. Rather than storing full transcripts, the episodic tier stores structured summaries: the nature of the dispute, the outcome, commitments made. At session start, Tier Two and Three content loads into context — and the agent begins informed, not blank.

⚠ Critical Mistake — Storing Raw PII in AI Memory
An agent stores full customer records — name, DOB, account number, transaction history — in a vector database hosted outside Australia. The bank's APRA data residency obligations require customer data to remain in-country. The AI memory store, built quickly by an engineering team unfamiliar with the regulatory requirement, has created a latent compliance breach at the architecture layer. It may not be discovered until the next APRA operational risk review.
✓ The Fix
Classify all memory content at write time: PII, Sensitive-Commercial, or Public. Store only tokenised references in the AI memory store — the foreign key to the customer record, not the record itself. Actual PII remains in your existing, governed, in-region data stores. The AI memory holds the pointer; your CRM or core banking system holds the data.

Structured Prompting

The engineering discipline of deterministic, governed instructions
Foundation Discipline

Structured Prompting is not about writing clever prompts. It is the engineering discipline of creating prompts that are versioned, testable, governed, and maintainable — so that every AI decision can be traced to a specific, approved instruction set.

The system prompt is the most consequential piece of configuration in an AI system. It determines what the agent will and will not do, how it reasons, what format it outputs, and what constraints it operates within. A change to the system prompt is a change to the AI's behaviour — and in a regulated institution, behaviour changes require governance.

The instruction hierarchy is the foundational concept: three levels of authority. Operator-level instructions (your system prompt) define the agent's purpose, persona, and hard constraints. User-level instructions are the incoming query. Retrieved context from RAG is the lowest authority — factual grounding that informs but does not override operator rules. This hierarchy must be explicit in the system prompt and enforced by the agent's behaviour.

Output schema enforcement — credit decision agent// System prompt instruction: "You must respond ONLY with a valid JSON object matching this schema. Never add explanatory prose outside the JSON structure. { 'decision': 'APPROVE' | 'DECLINE' | 'REFER' | 'ESCALATE', 'confidence': number between 0.0 and 1.0, 'reasoning_steps': string[] (one per evaluation criterion), 'regulatory_refs': string[] (policy/regulation citations), 'flags': string[] (risk flags requiring human attention), 'recommended_action': string (specific next step), 'expiry': ISO 8601 datetime }"
⚠ Critical Mistake — Unversioned Prompts in Application Code
The credit agent's system prompt lives as a string constant in a Spring Boot service. Four developers have access to the file. Over three months, three small changes are made with no change request raised, no approval sought, no version recorded. When the decline rate rises and the audit committee asks which prompt drove which decisions in which week — nobody can answer. This is a model governance failure under CPS 230.
✓ The Fix
Treat system prompts as versioned configuration artefacts stored in a configuration management system (AWS SSM, Azure App Configuration, or a dedicated prompt registry), tagged with semantic version numbers, requiring approval for changes affecting regulated use cases. Log the prompt version identifier alongside every inference decision.

Layer Two · Orchestration Patterns

A single capable agent is a reliable worker.
Ten agents without coordination is an organisational disaster.

How agents decompose complex work, route tasks to specialists, maintain quality, and process at scale — the patterns that transform a collection of agents into a system with genuine enterprise capability.

Enterprise Principle
Specialise agents — do not generalise them. Narrow agents with one job, clear I/O contracts, and bounded tool access are testable, replaceable, and auditable. These are the properties that matter in a regulated environment where you must explain every decision and replace any component independently.
07

Planner–Executor

Strategic decomposition meets governed execution
Enterprise Core

A Planner agent decomposes a high-level goal into a structured task graph. Executor agents carry out individual tasks, respecting dependencies, and report results back. The Planner synthesises outputs into a coherent final response. This is program management — applied to AI.

The Planner-Executor pattern maps directly to how enterprise architects think about large program decomposition. A business objective is translated into a capability map; capabilities decompose into work packages; work packages have dependencies that must be sequenced correctly. The Planner produces a DAG (Directed Acyclic Graph) and the execution engine honours the dependency ordering.

SME onboarding with trade finance — Planner output// Goal: "Onboard new SME customer and set up trade finance facility" Planner — task DAG: { tasks: [ { id:"T1", agent:"kyc_agent", depends_on:[], action:"verify_identity_documents" }, { id:"T2", agent:"aml_agent", depends_on:[], action:"build_aml_profile" }, { id:"T3", agent:"credit_agent", depends_on:["T1","T2"], action:"assess_facility_limit" }, { id:"T4", agent:"docs_agent", depends_on:["T3"], action:"generate_facility_agreement" }, { id:"T5", agent:"notify_agent",depends_on:["T4"], action:"send_welcome_package" } ] } // Execution engine honours DAG: T1 ‖ T2 → T3 → T4 → T5

The execution engine — critically — is not the Planner or any Executor. It is a dedicated workflow orchestrator: Temporal, AWS Step Functions, or equivalent. This distinction matters enormously. Using the LLM to manage its own execution order introduces non-determinism into a process that must be reliable, auditable, and recoverable on failure.

⚠ Critical Mistake — Task Plan as List, Not Graph
The Planner outputs a sequential list. The execution engine runs them in list order. The Planner, operating stochastically, places credit assessment (T3) before AML profiling (T2) in the list one time in twenty. The credit assessment executes using incomplete AML data. In banking, a credit decision without completed AML profiling is a regulatory breach under the AML/CTF Act 2006.
✓ The Fix
Model the task plan as a DAG with explicit dependency declarations. Validate the DAG structure before any execution begins — reject plans that contain cycles, reference unknown agents, or violate registered dependency rules. Use a workflow engine to enforce execution order at the infrastructure level. The workflow engine's execution log becomes your audit trail for task ordering compliance.

Supervisor–Worker

One front door, an orchestra of specialists behind it
Platform Pattern

The Supervisor–Worker pattern is the natural architecture for an AI banking platform. A single Supervisor agent receives every request, classifies intent, routes to the right specialist Worker, and aggregates the response. Workers know their domain deeply and nothing else.

The Supervisor acts as both a control plane and a customer experience layer. From the customer's perspective, there is one coherent AI system. Behind that interface, specialist Workers handle different domains — each with its own system prompt, tool set, RAG knowledge base, and quality thresholds. Workers operate in complete isolation from each other. This is not an architectural accident — it is a deliberate safety property. Workers cannot inadvertently influence each other, cannot create feedback loops, and can be tested, updated, and audited independently.

⚠ Critical Mistake — LLM Routing Without Confidence Fallback
On an ambiguous query — "what are the fees for international transfers?" — the classifier assigns 0.62 confidence but routes to the product Worker anyway. The product Worker lacks the international fee schedule, which lives in the payments Worker. The customer receives incomplete information delivered with AI confidence. Under ASIC obligations around fee disclosure for retail banking customers, this is not a minor accuracy issue.
✓ The Fix
Set an explicit confidence threshold — typically 0.75 for routing decisions in regulated contexts. Below threshold, route to a clarification agent whose sole purpose is to ask one disambiguation question and pass the clarified query back to the Supervisor. Log every routing decision: the query, classification result, confidence score, threshold applied, and Worker selected.

Critic–Refiner

Adversarial quality assurance at AI speed
Quality System

The Critic–Refiner pattern pairs a generator with an independent reviewer. Agent A produces a draft; Agent B evaluates it against a structured rubric and returns specific failures; Agent A revises. Human reviewers see not just the output, but the quality assurance log that validated it.

The Critic's rubric is the key artefact to design carefully. A good rubric is specific ("Does the DTI ratio calculation in section 3.2 match the components listed in Appendix A?"), not vague ("Is this accurate?"). It tests measurable properties, not subjective ones. And it produces structured output — not free-text feedback — so the Generator can act on it programmatically in the next revision cycle.

⚠ Critical Mistake — No Iteration Cap
A Critic–Refiner pair deployed for APRA regulatory return generation enters an infinite loop: the Critic always finds something imperfect; the Refiner always revises; the system never declares the output acceptable. The regulatory return deadline passes. The bank submits late. This is not a hypothetical — several organisations have reported exactly this failure mode in production deployments without iteration caps.
✓ The Fix
Define "done" with two conditions: all mandatory rubric criteria must pass AND model confidence must exceed 0.88. Set a hard iteration cap at three rounds. On cap breach — an explicit escalation to human review with the final draft and the full critique log attached. The human reviewer makes the final call, informed by the machine's quality assessment.

Map–Reduce

Distributed intelligence for documents that exceed any context window
Scale Pattern

Every large language model has a context window limit. Large enterprise documents — ISDA master agreements, portfolio review packs, multi-fund regulatory submissions — routinely exceed it. Map–Reduce processes them by splitting, parallelising, and aggregating.

In the Map phase, the document is split into independent chunks, each processed in parallel by a Map agent applying the same structured extraction task to its segment. In the Reduce phase, a Reducer agent receives all extraction results, deduplicates overlapping findings, resolves conflicts, and synthesises a unified output. For contract review in a capital markets context, this transforms a two-day analyst task into a twenty-minute automated extraction with human review at the end.

⚠ Critical Mistake — No Chunk Overlap at Boundaries
A Material Adverse Change clause begins on the last three lines of chunk 20 and continues through the first seven lines of chunk 21. Chunk 20's agent extracts an incomplete clause. Chunk 21's agent extracts a fragment that appears standalone. The Reducer treats them as two distinct clauses — creating a ghost clause and silently losing the complete MAC definition. This error surfaces only in litigation.
✓ The Fix
Implement 100–200 token overlap at every chunk boundary. Include explicit conflict detection in the Reducer prompt: "If two adjacent chunk extractions appear to describe the same clause, combine them. If two non-adjacent chunks produce conflicting values for the same term, flag both as CONFLICT and surface both versions for human review." Never allow the Reducer to silently resolve conflicts.

11

Tool / Function Calling

The bridge between language and action
Core Integration

Tool calling is the mechanism by which an AI agent interacts with the real world. The model produces structured arguments; your runtime executes the actual call. What's in the tool registry determines everything an agent can do — and everything it can break.

A well-designed tool registry is a governed service catalogue with explicit security properties. Every tool entry must carry: a name and description precise enough to route correctly without human intervention; a typed, validated parameter schema; the OAuth scope required; a data classification tag so every call is logged with appropriate sensitivity; and a rate limit to prevent a looping agent from triggering cascading API failures downstream.

The description is the most important field. The model reads tool descriptions to decide which tool to call. Writing tool descriptions as you would write API documentation for a thoughtful junior developer — explaining what it does, what it returns, when to use it versus alternatives, and what errors to expect — is the single highest-leverage improvement available in most agentic systems.

⚠ Critical Mistake — Agent Has Write Access It Doesn't Need
The tool registry includes update_customer_record alongside read-only lookup tools, because a developer copied the tool set from an internal admin API. A prompt injection in a customer-provided free-text field causes the agent to call the write tool with adversary-controlled parameters. The result: corrupted customer data in core banking — simultaneously a data integrity incident and a CPS 234 information security incident, both requiring regulatory notification.
✓ The Fix
Apply principle of least privilege to every agent's tool set at design time. Customer-facing read-only agents receive only read-only tools. Write operations require explicit human confirmation — implemented as a HITL gate in the workflow, not as an agent decision. Audit every tool call with the agent's session context. Review tool usage logs weekly for anomalous patterns.

Event-Driven AI

Real-time intelligence on the payments rail
Real-Time Critical

Event-Driven AI activates agents in response to business events, not user requests. It is the architecture that places AI intelligence directly on the payment processing path — making fraud prevention a real-time intervention, not a retrospective review.

In the NPP (New Payments Platform) context, a payment event arrives on a Kafka topic. A pre-screening rule engine — operating in microseconds — evaluates the event against defined risk signals. When three or more signals are present, the fraud analysis agent is invoked with the full event payload. The agent enriches with device history, payee velocity, behavioural biometrics, and geo-anomaly scoring. A risk verdict and recommended action are returned within the NPP's five-second processing window.

Event-driven fraud scoring — NPP paymentEvent: Kafka: payments.npp.outbound { amount:45000, beneficiary:"NEW_PAYEE", time:"02:14", device:"NEW_MOBILE" } Pre-screen: 3 risk signals — agent activation threshold met Enrichment: device_history: 0 prior transactions on this device payee_velocity: new payee, 0 prior payments geo_anomaly: login Melbourne; typical: Sydney behavioural_score: 0.91 deviation from baseline Output: { risk:0.94, action:"HOLD_SOFT", reason:"4 concurrent risk signals" } Timeline: 680ms end-to-end · NPP window: 5000ms
⚠ Critical Mistake — LLM Latency Exceeds the Payment SLA Window
LLM inference plus tool calls plus reranking takes 6–9 seconds. Under peak load, payments begin timing out. The gateway, receiving no fraud decision within the NPP window, fails safe — declining legitimate payments. The bank receives 2,400 complaints in four hours. This is a service reliability incident under NPP Connected Institution obligations.
✓ The Fix
Tiered, latency-gated scoring: Tier 1 (≤100ms) deterministic rule engine. Tier 2 (≤800ms) lightweight ML scoring model, handles 90% of transactions. Tier 3 (async) LLM deep analysis for the 5–10% in the mid-risk band. Only Tier 1 and 2 are on the synchronous payment path. LLM intelligence enriches retrospective case management and improves Tier 2 model training.

Semantic Cache

Instant answers for questions that have already been answered
Performance

60–80% of FAQ traffic at a retail bank is semantically equivalent — the same questions phrased differently. Semantic Cache stores AI responses keyed by meaning rather than exact text, delivering sub-10ms responses for common queries while reserving full inference for genuinely novel ones.

Standard HTTP caching fails for natural language because "what is my home loan rate?" and "can you tell me the interest rate on my mortgage?" have different strings but identical intent. Semantic Cache solves this by embedding incoming queries as vectors and retrieving cached responses by cosine similarity. Above a threshold (typically 0.92–0.95), the cached response is returned immediately. Below threshold, the full agent pipeline runs, and the result is cached for subsequent similar queries.

⚠ Critical Mistake — Caching Personalised Responses
"What is MY account balance?" is semantically very similar to "What's my current balance?" — both embed close in vector space. If the cache does not correctly identify this as personalised and cache-ineligible: Customer A's balance of $42,318 gets cached. Customer B, fifteen minutes later, asks "What's my current balance?" The semantic similarity is 0.96 — above threshold. The cache returns Customer A's balance to Customer B. This is a privacy breach notifiable to the OAIC under the Privacy Act 1988 Notifiable Data Breaches scheme.
✓ The Fix
Implement a query classifier that runs before cache lookup. Any query containing personalisation signals (my, your, account, balance, transaction, statement) or real-time signals (current, today, now, latest) is immediately classified as cache-ineligible and routed to the full agent pipeline. Only generic informational queries — rates, policy terms, product features, locations — enter the semantic cache. Enforce this classification at the gateway level.

Human-in-the-Loop

The blocking gate that makes AI safe for consequential decisions
Governance Gate

Human-in-the-Loop is not a checkbox, a notification, or a "flag for review" button. It is a blocking gate embedded in the workflow DAG — a structural pause before consequential actions that requires a human decision before the agent may proceed.

The pattern exists because some decisions carry consequences that cannot be reversed by a subsequent correction. A payment made incorrectly may be recoverable; reputational damage to a customer who received an erroneous credit decline is not. HITL gates prevent the class of errors that are easier to avoid than to fix. In regulated Australian financial services, minimum escalation criteria include: agent confidence below 0.75 on any financial decision; transaction value exceeding the pre-approved automation limit; customer with an active financial hardship flag; any PEP, sanctions, or adverse media match; any action triggering a regulated disclosure obligation under IDR or EDR frameworks; and any explicit customer request for human assistance.

⚠ Critical Mistake — Retrospective Review Instead of Blocking Gate
The team adds a "flag for review" dashboard at the end. The agent sends emails, updates records, and submits forms — then flags high-risk cases for review the next morning. When the compliance reviewer checks, the agent has already acted. For a customer who received an incorrect hardship assessment email, the harm occurred the previous afternoon. Under ASIC's RG 271, the bank must now respond to a complaint about an error the system made before any human saw the output. The dashboard is not HITL. It is audit logging with a pleasant interface.
✓ The Fix
Design HITL as a first-class component in the workflow DAG from the initial architecture session. The agent produces a recommendation; the workflow pauses; a task is created in the human review queue; the agent does nothing further until the reviewer approves or modifies. Use Temporal, AWS Step Functions, or equivalent to enforce this structurally — not a boolean flag in application code that a future developer might disable for performance reasons.

The operational wrapper that transforms a prototype into a production system. In regulated financial services, this layer is not optional — it is the entire argument for deployment at scale.

Enterprise Principle

Your framework must address model change management, data lineage, incident response, vendor concentration risk, and third-party assurance for model providers.
15

Guardrails Framework

Input and output safety enforcement at the infrastructure layer
Safety · Mandatory

Guardrails intercept and validate both inputs to agents and outputs from them. They are not a user experience feature. They are a security control — and they must be applied with the same universality and rigour as other security controls in your technology estate.

Input guardrails run before the agent processes any request: prompt injection detection (regex + LLM classifier), PII redaction before reaching an external LLM API, topic scope enforcement, and token budget limits. Output guardrails run before the response reaches the caller: faithfulness checking against retrieved context, PII leakage scanning, regulatory accuracy checking for rate or policy claims, and format compliance validation against the declared schema.

⚠ Critical Mistake — Guardrails Only on Customer-Facing Agents
Rigorous guardrails are applied to customer-facing agents and none to internal staff tools, reasoning that "staff won't try to exploit them." This fails on three vectors: insider threat (a disgruntled employee uses prompt injection to extract unauthorised customer records), compromised credentials (an external attacker with stolen staff login uses the internal tool to exfiltrate data at scale), and accidental misuse (a well-intentioned staff member inadvertently surfaces data across customer boundaries). All three produce the same result: an OAIC notifiable data breach.
✓ The Fix
Apply guardrails uniformly across all agents with no exceptions for internal tools. Maintain separate guardrail configurations per agent class (customer-facing, staff-facing, automated pipeline) — but maintain them, version them, and audit them all. Guardrail health is a metric in your AI observability dashboard alongside throughput and latency.

Circuit Breaker

Intelligent failure isolation with meaningful fallback
Resilience

When an AI service degrades, the system should route to a fallback that still solves the customer's problem — not to a generic error. A circuit breaker without a meaningful fallback is simply a way of failing faster, which is not the same as failing gracefully.

Three states: Closed (normal — all traffic to AI), Open (degraded — triggered when error rate exceeds 5% over 60 seconds or P95 latency exceeds the SLA threshold — all traffic to fallback), and Half-Open (recovery probe — after 30 seconds, 10% of traffic to AI while 90% continues to fallback; if probe normalises, return to Closed). The fallback strategy is the most important design decision — and the most frequently neglected. A fallback must be designed with the same customer-centricity as the primary AI path.

⚠ Critical Mistake — Fallback Routes to a Generic Error
The AI banking assistant's circuit breaker opens during a peak period (9am Monday). The fallback renders: "Our AI assistant is temporarily unavailable. Please try again later." During the 4-hour degradation period, 47,000 customers encounter this message. The majority call the contact centre. Call volume spikes 340%. Average wait time reaches 28 minutes. The AI outage has created a contact centre capacity crisis. The circuit breaker did exactly what it was designed to do — and made everything worse, because the fallback was not designed.
✓ The Fix
Design the fallback as a first-class product experience, developed in parallel with the AI path. For customer-facing agents, the fallback must be functional, not apologetic. Test the fallback monthly in production — deliberately trigger the circuit breaker in a non-peak window and verify that customer journeys complete on the fallback path.

AI Observability

The audit trail that answers the regulator's question
Regulatory Baseline

Standard APM tells you the service was slow. AI Observability tells you why the agent decided what it decided — with enough fidelity to reconstruct any decision from any point in time, months or years later, under regulatory examination.

AI Observability goes beyond traditional APM in three critical dimensions: it captures model-specific signals (prompt version, model identifier, token distribution, confidence score, guardrail evaluation results); it maintains full inference context (not just the output, but the complete input including system prompt, user query, and all RAG chunks injected); and it tracks decision drift over time — detecting when the distribution of decisions shifts in ways that indicate model degradation, prompt version changes, or retrieval quality deterioration.

⚠ Critical Mistake — Logging Output but Not Prompt Context
Logs record the agent's decision — DECLINED — and the timestamp. When an APRA review examines a sample of automated credit decisions, the team can show what was decided and when. They cannot show which prompt version governed the decision, which RAG chunks were in context, what the confidence was, or what tool calls occurred. The decision cannot be reconstructed. Under CPS 230, this is a model risk management finding that triggers remediation requirements.
✓ The Fix
Log the complete inference context as a single atomic record at every agent decision point. Store in an immutable, append-only log store with tamper-evident properties (AWS S3 with Object Lock, or Azure Blob with immutability policies). Apply retention class tagging at write time: regulated credit and AML decisions require seven-year retention in Australia.

Cost & Token Governance

The FinOps layer that keeps AI economically sustainable
FinOps

Two million AI interactions per day at the wrong model tier generates a multi-million dollar monthly inference bill before business value is proven. Cost and Token Governance is the pattern that maps intelligence to price — routing each task to the cheapest model that meets the quality requirement.

Three tiers, classified at design time based on task complexity benchmarks: Small models (Haiku-class) for simple lookups, FAQ, and triage classification — fastest, cheapest, achieves 92–96% of larger model quality on these tasks. Mid-tier models (Sonnet-class) for document analysis, drafting, and structured extraction — balanced quality and cost. Full models (Opus-class) for complex reasoning, regulatory analysis, and high-stakes decisioning — the cost premium is justified by the quality gap.

Budget enforcement operates at three levels: per-session token limits (preventing runaway agentic loops), per-agent daily budgets (preventing any single use case from consuming disproportionate capacity), and per-business-unit weekly reporting (making AI cost visible to the teams generating it — which consistently produces more thoughtful usage than invisible shared infrastructure).

⚠ Critical Mistake — Largest Model for Every Task
The AI platform team, prioritising quality and risk-aversion, configures the largest available model as the default for every agent. Early pilots at 50,000 interactions/month show no cost problem. The programme scales to 1.5 million interactions/month. The monthly inference cost is presented to the CFO. The programme is suspended pending a cost-benefit review. Three months of suspension adds $2M in programme management cost and delays delivery by a quarter. The pattern that would have prevented this — model tier routing based on task complexity — requires approximately two days of benchmarking work.
✓ The Fix
Conduct task-complexity benchmarking before deployment. For each use case, run 200–500 representative queries through all available model tiers. Measure quality (accuracy, format compliance, safety metrics) and latency. Identify the cheapest model tier that meets your quality threshold for each use case. Implement model tier routing in the API gateway layer — not in individual agent code. Review routing allocations quarterly as model capabilities and prices evolve.
